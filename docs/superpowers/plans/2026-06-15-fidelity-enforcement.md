## Goal (the bar this must hit)

Resume and cover letter must be generated by reasoning over the candidate's RAW resume,
exactly like pasting it into the Claude web UI, using the same model (PREMIUM_MODEL =
Opus). Content quality = web UI. Formatting stays in the existing deterministic template
(a feature: consistent layout + two-page control). No fabricated employers. Correct years
of experience. "Consistent" means reliably high quality, not bitwise-identical output (the
web UI is not identical run to run either).

## Locked product decisions (2026-06-15, do not revisit)

- Raw resume (`resumeRawText`) is the SINGLE SOURCE OF TRUTH for all generated content.
- The structured bank is DERIVED DATA used only for mechanics: entry ids, ordering,
  the casual flag, role/company/date headers, and skills. It is never the content source.
- The PROFILE EDITING FEATURE IS REMOVED ENTIRELY. Users change their data by re-uploading
  their resume (existing `forceAutoExtract` re-extraction path). Because nothing can edit
  the bank independently, there is NO edit-to-rawText sync to build.

## Context already true in the code (do not rebuild)

- `resumeRawText` is stored (`schema.prisma:32`) and confirmed populated for live profiles.
- Cover letter already reasons from the raw resume as its single source of truth
  (`coverLetterSlotsPrompt.ts:45,55-57`). No change needed there.
- Resume currently does the OPPOSITE: it treats the structured bank as authoritative and
  the raw resume as "reference only" (`resumeStructuredPrompt.ts:81-89,104`). Stage 3 flips
  this. This is the core parity bug.
- Years is computed (`lib/profileMath.ts`), stored (`autoExtract.ts:368-377`, `>=2` floor),
  and passed to the cover-letter prompt with a hard guard (`generate.ts:841`,
  `coverLetterSlotsPrompt.ts:34-36`).
- Extraction prompt already forbids inventing employers (`extraction.ts:5`) but the model
  sometimes disobeys (Noble Seeds). Stage 1 enforces it in code.
- The resume renderer (`lib/buildTemplateResume.ts`) pulls each role's company/role/dates
  from the stored Experience rows by id; the LLM returns only bullets/order/flags. So a
  fabricated company in the bank renders on the resume even if the LLM ignores it — which
  is exactly why Stage 1 (clean the bank) is required in addition to Stage 3.

## Stage 5 — Deterministic output backstops

DESIGN RULE: this backstop NEVER mutates generated prose by blanking or splicing a phrase
(that produces broken sentences like "over the past ____, I have developed..."). It detects
a violation and REGENERATES; only as an absolute last resort does it drop a whole sentence.

5.1 New `server/src/lib/yearsClaimDetector.ts`: `detectYearsClaim(text): { found, value }`.
    Detect digit and written-number experience claims (`/\b(\d{1,2})\s*\+?\s*years?\b/i` and
    one..twelve followed by "years", tied to experience). Pure detection, no mutation.
5.2 Wire a check after generation (cover-letter path after `enforceCoverLetterQuality`
    `generate.ts:877`; resume summary post-processing), passing
    `storedYears = profile.yearsOfExperience ?? null`. A claim VIOLATES when: storedYears is
    null/<2 and any number is present, OR storedYears is set and the stated number != storedYears.
5.3 On violation: re-run the SAME single-pass generation ONCE via the existing
    `analysisContext.regenerateFeedback` channel, injecting the Claude-owned instruction
    (verbatim):
    - storedYears null/<2: "Do not state any number of years of professional experience
      anywhere in the document. Lead with the qualification and the nature of the experience
      instead."
    - storedYears = N: "State the candidate's professional experience as exactly N years
      wherever length of experience is mentioned. Do not state any other figure."
    Re-check the regenerated output. If it STILL violates (rare), remove only the complete
    sentence(s) containing the claim — never blank a phrase. Guard against infinite loops:
    at most one regeneration.
5.4 Tests: detector finds "over the past four years" and "seven years of experience";
    given storedYears=null a violating output triggers exactly one regeneration with the
    no-number instruction; given storedYears=5 a "seven years" output triggers one
    regeneration with the exactly-5 instruction; a compliant output triggers none.
5.5 (Report only) Employer grounding check on generated cover-letter text: flag capitalized
    multi-word employer-like phrases whose tokens are absent from the grounded employer set.
    Log only; STOP and report the flag list before any auto-removal.
