 Job Feed Performance Investigation

  Engineering Summary for Senior Developer Review

  Date: June 19, 2026

  ---
  1. The Problem

  The job feed has a ~26 second latency bottleneck when generating bullets (job summary points) for feed     
  items. This was blocking the entire feed response, causing users to stare at a loading spinner.

  Key metrics from logs:
  - Scraping 3 roles in parallel: ~fast (hundreds of ms)
  - Bullet generation: ~26 seconds (synchronous, blocking)
  - Total feed build: variable, but bullets dominate

  3. What We Did (Current Work)

  A. Async Bullet Generation (In Progress)

  Moved bullet generation from blocking to fire-and-forget:

  Before (blocking):
  // OLD: Server waited ~26s for LLM before responding
  const bullets = await generateBullets(jobs);
  await prisma.jobFeedItem.createMany({ data: jobs.map((j, i) => ({ ...j, bullets: bullets[i] })) });        
  return res.json({ jobs });

  After (async):
  // NEW: Return feed immediately, kickoff bullets in background
  const items = await prisma.jobFeedItem.findMany({ ... });
  const nullBulletItems = items.filter(i => i.bullets === null);
  if (nullBulletItems.length > 0) {
    kickoffBulletGeneration(nullBulletItems); // Fire-and-forget
  }
  return res.json({ jobs: items, bulletsPending: nullBulletItems.length > 0 });

  Key mechanisms:
  - bulletGenInFlight Set prevents duplicate generation on repeated polls
  - Client polls /feed while bulletsPending === true
  - Bullets write to DB as they land; next poll picks them up

  5. The Remaining Bottleneck

  Bullet generation is still ~26 seconds — we've just moved it off the critical path. The root cause:        

  // server/src/services/jobFeed.ts:67-97
  export async function generateBullets(jobs: RawJob[]): Promise<(string[] | null)[]> {
    const batch = jobs.slice(0, 10);
    const prompt = `For each job below, write exactly 3-5 bullet points...
   
    Return a JSON array of arrays: [[bullet, bullet, ...], ...]`;

    const raw = await callLLMWithRetry(prompt, true); // ← 26s here
    // ...
  }

  Why it's slow:
  - Single LLM call processing 10 jobs × 3-5 bullets each
  - Unknown model (likely callLLMWithRetry default)
  - No streaming, must complete before any bullets available

  ---
  6. Next Steps / Recommendations

  Immediate (Performance)

  1. Parallel bullet generation — One LLM call per job instead of batch
    - Trade-off: More API calls, but 10× parallel = ~3s instead of ~26s
    - Or: Streaming JSON parser to emit bullets as they arrive
  2. Model selection — Verify callLLMWithRetry uses Haiku for this (fast), not Opus (slow)
  3. Caching bullets — Cache bullet arrays by (job title + company + first 100 chars of description) hash so 
  identical jobs don't re-generate

  Medium-term

  4. Pre-generation — Generate bullets at ingestion time (cron), not user-request time
  5. Seek full description — Currently hydrating on-demand; consider fetching full description at scrape time  (cost trade-off: $1/1k jobs)
